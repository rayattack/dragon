# The Zen of Dragon

Dragon is - the snake that became a Dragon.
Here, the hot path is sacred.
If interpreted it is, sacred it is not.

Fast is better than slow.
Slow is better than broken.
You should know if it is broken when you compile.
Practicality should not be an excuse for poor performance.

If you are the only one that understands it, it is wrong.
In the face of ambiguity, annotate.
If it walks like a duck, declare it `Duck`.
Await is a verb, not a label.

Errors are feedback you might not like, not mistakes.
Half-measures are full mistakes.
A silent fallback is a silent lie.
A workaround is a wound - fix the root cause.
and tests that pass for the wrong reason are bugs in disguise.
If it can be done in Dragon, it must be done in Dragon.

Nothing should ever be complex.
But complex is better than nothing.
A scope that ends frees what it held.
Readability counts; the benchmark counts the votes.
There should be one-- and preferably only one --obvious way to do it.
If it isn't the fastest way, you haven't found it yet.
If you cannot explain the lowering, you cannot trust the speed.

Types are one honking great idea -- let's monomorphize more of those!
