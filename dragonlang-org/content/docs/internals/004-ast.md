# 004 -- Dragon AST: The Abstract Syntax Tree

> **Source files:** `include/dragon/AST.h`, `src/AST.cpp`
> **Last Updated:** 2026-06-22
> **Test suite:** ASTTests (36 tests)

---

## 1. Overview

Dragon's Abstract Syntax Tree is the central data structure of the compiler. Every stage after the parser operates on it: semantic analysis reads it, the type checker annotates it, and the code emitters traverse it to produce output.

The AST is defined in `include/dragon/AST.h` (node class declarations and visitor interface) and `src/AST.cpp` (accept method implementations, DefaultASTVisitor, and ASTPrinter).

### Node counts

The AST contains **62 concrete node classes** organized into five categories:

| Category | Count | Base class |
|----------|-------|------------|
| Type expression nodes | 6 | `TypeExpr` |
| Expression nodes | 30 | `Expr` |
| Statement nodes | 22 | `Stmt` |
| Declaration nodes | 3 | `Stmt` (treated as statements) |
| Module | 1 | `ASTNode` (direct) |
| **Total** | **62 visitable types** | |

Note: `FunctionDecl`, `ClassDecl`, and `TypeAliasStmt` extend `Stmt`, not a separate `Decl` base class. This means they can appear anywhere a statement is expected. The `Module` node extends `ASTNode` directly. The three abstract bases (`Expr`, `Stmt`, `TypeExpr`) also derive from `ASTNode` but are not visitable node types of their own.

### `ASTNode` -- The root

```cpp
class ASTNode {
public:
    virtual ~ASTNode() = default;
    virtual void accept(ASTVisitor& visitor) = 0;

    SourceLocation location() const { return location_; }
    void setLocation(SourceLocation loc) { location_ = loc; }

protected:
    SourceLocation location_;
};
```

Every AST node has:
- A pure virtual `accept()` method for the visitor pattern
- A `SourceLocation` tracking where in the source file this node originated
- A virtual destructor for safe polymorphic deletion

### `Expr` -- Expression base

```cpp
class Expr : public ASTNode {
public:
    std::shared_ptr<Type> type;   // Set by TypeChecker, nullptr until then
};
```

Extends `ASTNode` with a single field: `type`. This is a `shared_ptr<Type>` that is initially null and gets set during type checking. The `Type` class hierarchy (defined in `include/dragon/TypeChecker.h`) includes `PrimitiveType`, `ListType`, `DictType`, `TupleType`, `FunctionType`, `ClassType`, `InstanceType`, `UnionType`, `AnyType`, `NeverType`, and `UnknownType`.

### `Stmt` -- Statement base

```cpp
class Stmt : public ASTNode {};
```

A marker class with no additional fields beyond what `ASTNode` provides.

### `TypeExpr` -- Type annotation base

```cpp
class TypeExpr : public ASTNode {};
```

A marker class for type annotations. Type expressions represent the syntactic form of types as written in source code (e.g., `list[int]`), as opposed to the semantic `Type` objects resolved by the type checker.

## 3. Source Location Tracking

Every AST node inherits a `SourceLocation` from `ASTNode`:

```cpp
struct SourceLocation {
    std::string filename;   // e.g., "main.dr", "<stdin>"
    size_t line = 0;        // 1-based line number (0 = unset)
    size_t column = 0;      // 1-based column number (0 = unset)
    size_t offset = 0;      // 0-based byte offset from start of file
};
```

The location is set by the parser when creating nodes, using `node->setLocation(token.location())`. Note that `filename` is stored as `std::string` (not `string_view`) to avoid lifetime issues -- the token stream may be destroyed before the AST is finished being used.

Not all nodes have their location explicitly set by the parser. Nodes like `BreakStmt`, `ContinueStmt`, and `PassStmt` are created without calling `setLocation()`, so their location fields remain at defaults (`line = 0, column = 0`). Nodes where the parser does set the location include: `IntegerLiteral`, `FloatLiteral`, `StringLiteral`, `BooleanLiteral`, `NoneLiteral`, `NameExpr`, `FunctionDecl`, `ClassDecl`, and `ReturnStmt`.

## 4. Type Expression Nodes (6 nodes)

Type expression nodes represent type annotations as they appear in source code.

### `NamedTypeExpr` -- Simple type name

```cpp
class NamedTypeExpr : public TypeExpr {
public:
    std::string name;   // e.g., "int", "str", "MyClass", "None"
};
```

Represents a bare type name. Also used for `None` as a type (the parser creates `NamedTypeExpr("None")` when it sees the `NONE` token in type position).

**Examples:** `int`, `str`, `float`, `bool`, `None`, `MyClass`

### `GenericTypeExpr` -- Parameterized type

```cpp
class GenericTypeExpr : public TypeExpr {
public:
    std::unique_ptr<TypeExpr> base;                     // The base type (e.g., NamedTypeExpr("list"))
    std::vector<std::unique_ptr<TypeExpr>> typeArgs;    // The type parameters (e.g., [NamedTypeExpr("int")])
};
```

Represents a type with type parameters in brackets.

**Examples:** `list[int]`, `dict[str, int]`, `Optional[str]`, `set[float]`

### `OptionalTypeExpr` -- Optional type

```cpp
class OptionalTypeExpr : public TypeExpr {
public:
    std::unique_ptr<TypeExpr> inner;   // The wrapped type
};
```

Represents an optional type. Note that in the current parser, `Optional[int]` is parsed as a `GenericTypeExpr` (base=`Optional`, typeArgs=[`int`]), and `int | None` is parsed as a `UnionTypeExpr`. The `OptionalTypeExpr` node exists in the AST definition but is not currently produced by the parser -- it is available for future use or for semantic lowering.

### `UnionTypeExpr` -- Union type

```cpp
class UnionTypeExpr : public TypeExpr {
public:
    std::vector<std::unique_ptr<TypeExpr>> types;   // All member types
};
```

Represents a union of multiple types using the `|` syntax.

**Examples:** `int | str`, `int | str | None`, `list[int] | dict[str, int]`

### `CallableTypeExpr` -- Callable/function type

```cpp
class CallableTypeExpr : public TypeExpr {
public:
    std::vector<std::unique_ptr<TypeExpr>> paramTypes;   // Parameter types
    std::unique_ptr<TypeExpr> returnType;                 // Return type
};
```

Represents `Callable[[param_types], return_type]` syntax. Parsed when the parser sees `Callable[[` (double bracket).

**Example:** `Callable[[int, str], bool]` represents a function taking `int` and `str` and returning `bool`.

### `TupleTypeExpr` -- Tuple type

```cpp
class TupleTypeExpr : public TypeExpr {
public:
    std::vector<std::unique_ptr<TypeExpr>> elementTypes;   // Type of each element
};
```

Represents `tuple[type1, type2, ...]` syntax. Note that in the current parser, `tuple[int, str]` is parsed as a `GenericTypeExpr` (base=`tuple`, typeArgs=[`int`, `str`]). The `TupleTypeExpr` node exists for potential semantic lowering or future parser enhancements.

## 5. Expression Nodes (30 nodes)

All expression nodes extend `Expr` and inherit the `shared_ptr<Type> type` field.

### `IntegerLiteral` -- Integer constants

```cpp
class IntegerLiteral : public Expr {
public:
    int64_t value;   // The parsed integer value
};
```

Represents integer literals in decimal, hex, binary, or octal. The parser handles base detection and `_` separator stripping.

**Examples:** `42`, `0xFF`, `0b1010`, `0o77`, `1_000_000`

### `FloatLiteral` -- Floating-point constants

```cpp
class FloatLiteral : public Expr {
public:
    double value;   // The parsed float value
};
```

**Examples:** `3.14`, `1e-10`, `1_000.5`

### `StringLiteral` -- String constants

```cpp
class StringLiteral : public Expr {
public:
    std::string value;     // The string content (delimiters stripped)
    bool isRaw = false;    // r"..." or R"..."
    bool isFString = false; // f"..." or F"..."
    bool isBytes = false;   // b"..." or B"..."
};
```

Stores the string content with quote delimiters removed. Prefix flags indicate the string type. Triple-quoted strings (`"""..."""`, `'''...'''`) have their 3-character delimiters stripped.

**Examples:** `"hello"`, `'world'`, `f"x={x}"`, `r"\n"`, `b"bytes"`, `"""multiline"""`

For f-strings (`isFString=true`), the parser populates `fstringParts` - a `vector<FStringPart>` where each part is either a literal run of text or an interpolated expression (with optional `!conversion` and `:format_spec`). Sema, the TypeChecker, and CodeGen all walk this one shared AST instead of re-lexing the raw `value` at each stage.

### `TemplateExpr` -- Inline template literal

```cpp
class TemplateExpr : public Expr {
public:
    std::string body;          // raw template text (between outer braces)
    std::string contentType;   // empty = untyped (str), non-empty = typed template
    bool isContentAlias = false;
};
```

Represents an inline `template { raw text with !{expr} interpolation }` literal. The typed variant `template[HTML] { ... }` records its content type in `contentType` (empty means an untyped `str` template). The `isContentAlias` flag marks a `:{ ... }` content-alias fragment (spec-17 Phase 4.B), which inherits its `contentType` from the enclosing `template[X]` and auto-appends its rendered string to the enclosing block buffer.

**Examples:** `template { Hello !{name} }`, `template[HTML] { <p>!{body}</p> }`

### `TemplateFileExpr` -- Compile-time file template

```cpp
class TemplateFileExpr : public Expr {
public:
    std::string filePath;     // path to template file (string literal)
    std::string contentType;  // empty = untyped (str), non-empty = typed template
};
```

Represents `template("file.html")`, where the template body is read from a file at compile time. The typed variant `template[HTML]("file.html")` records its content type in `contentType`.

**Examples:** `template("page.txt")`, `template[HTML]("index.html")`

### `BooleanLiteral` -- Boolean constants

```cpp
class BooleanLiteral : public Expr {
public:
    bool value;   // true for True, false for False
};
```

### `NoneLiteral` -- The None value

```cpp
class NoneLiteral : public Expr {
    // No additional fields
};
```

### `NameExpr` -- Variable references

```cpp
class NameExpr : public Expr {
public:
    std::string name;   // The identifier text
};
```

Represents any unqualified name reference: variables, function names, class names, module names.

**Examples:** `x`, `my_var`, `print`, `MyClass`

### `BinaryExpr` -- Binary operations

```cpp
class BinaryExpr : public Expr {
public:
    std::unique_ptr<Expr> left;    // Left operand
    Token op;                       // The operator token (carries type, lexeme, location)
    std::unique_ptr<Expr> right;   // Right operand
};
```

Represents all binary operations. The `op` field is a full `Token` object, preserving the operator's lexeme and source location. The operator type is determined by `op.type()`:

| Operator | Token Type | Precedence Level |
|----------|-----------|-----------------|
| `or` | `OR` | orExpr |
| `and` | `AND` | andExpr |
| `<`, `>`, `<=`, `>=`, `==`, `!=` | `LESS`, `GREATER`, etc. | comparison |
| `in` | `IN` | comparison |
| `is` | `IS` | comparison |
| `\|` | `PIPE` | bitwiseOr |
| `^` | `CARET` | bitwiseXor |
| `&` | `AMPERSAND` | bitwiseAnd |
| `<<`, `>>` | `LEFT_SHIFT`, `RIGHT_SHIFT` | shift |
| `+`, `-` | `PLUS`, `MINUS` | term |
| `*`, `/`, `%`, `//` | `STAR`, `SLASH`, `PERCENT`, `DOUBLE_SLASH` | factor |
| `**` | `POWER` | power |

### `ChainedCompExpr` -- Chained comparison

```cpp
class ChainedCompExpr : public Expr {
public:
    std::vector<std::unique_ptr<Expr>> operands;   // a, b, c, ...
    std::vector<Token> operators;                   // <, <, ...
};
```

Represents a chained comparison like `a < b < c` or `0 <= x < 100`. The `operands` vector holds the N expressions and `operators` holds the N-1 comparison tokens between them. This is distinct from nesting `BinaryExpr` because Python/Dragon chained comparisons evaluate each operand once and short-circuit.

**Examples:** `a < b < c`, `0 <= x < 100`

### `WalrusExpr` -- Walrus (assignment) expression

```cpp
class WalrusExpr : public Expr {
public:
    std::string name;
    std::unique_ptr<Expr> value;
};
```

Represents the `:=` named-expression operator (PEP 572): it binds `name` to `value` and evaluates to that value. This is an implicit binding form (it declares with an inferred type, no annotation slot).

**Examples:** `(n := len(items))`, `while (chunk := read()) { ... }`

### `UnaryExpr` -- Unary operations

```cpp
class UnaryExpr : public Expr {
public:
    Token op;                          // The operator token
    std::unique_ptr<Expr> operand;     // The operand
};
```

Handles both prefix arithmetic (`-x`, `+x`, `~x`) and logical negation (`not x`). The operator type is in `op.type()`: `MINUS`, `PLUS`, `TILDE`, or `NOT`.

### `CallExpr` -- Function calls

```cpp
class CallExpr : public Expr {
public:
    std::unique_ptr<Expr> callee;                                           // The function being called
    std::vector<std::unique_ptr<Expr>> args;                                // Positional arguments
    std::vector<std::pair<std::string, std::unique_ptr<Expr>>> kwArgs;      // Keyword arguments (name, value)
};
```

The callee can be any expression: `NameExpr` for simple calls (`foo(x)`), `AttributeExpr` for method calls (`obj.method(x)`), another `CallExpr` for chained calls (`foo()(x)`).

**Examples:** `print("hello")`, `obj.method(x, key=val)`, `list(range(10))`

### `AttributeExpr` -- Attribute access

```cpp
class AttributeExpr : public Expr {
public:
    std::unique_ptr<Expr> object;      // The object being accessed
    std::string attribute;              // The attribute name
};
```

**Examples:** `obj.attr`, `self.name`, `module.Class`

### `SubscriptExpr` -- Subscript/indexing

```cpp
class SubscriptExpr : public Expr {
public:
    std::unique_ptr<Expr> object;   // The container being indexed
    std::unique_ptr<Expr> index;    // The index expression (or a SliceExpr)
};
```

The `index` field can be any expression for regular indexing, or a `SliceExpr` for slice operations.

**Examples:** `arr[0]`, `dict["key"]`, `matrix[i]`, `arr[1:5]` (index is SliceExpr)

### `SliceExpr` -- Slice notation

```cpp
class SliceExpr : public Expr {
public:
    std::unique_ptr<Expr> lower;   // Start index (optional, nullptr = from beginning)
    std::unique_ptr<Expr> upper;   // Stop index (optional, nullptr = to end)
    std::unique_ptr<Expr> step;    // Step value (optional, nullptr = step of 1)
};
```

Represents slice notation. All three fields are optional. A `SliceExpr` is always nested inside a `SubscriptExpr` as the `index` field.

**Examples:**
- `arr[1:5]` -- lower=1, upper=5, step=null
- `arr[::2]` -- lower=null, upper=null, step=2
- `arr[::-1]` -- lower=null, upper=null, step=-1
- `arr[1:]` -- lower=1, upper=null, step=null

### `ListExpr` -- List literals

```cpp
class ListExpr : public Expr {
public:
    std::vector<std::unique_ptr<Expr>> elements;   // The list elements
};
```

An empty `ListExpr` (no elements) represents `[]`.

**Examples:** `[1, 2, 3]`, `["a", "b"]`, `[]`

### `TupleExpr` -- Tuple literals

```cpp
class TupleExpr : public Expr {
public:
    std::vector<std::unique_ptr<Expr>> elements;   // The tuple elements
};
```

An empty `TupleExpr` represents `()`. A single-element tuple like `(x,)` has one element.

**Examples:** `(1, 2, 3)`, `(x,)`, `()`

### `DictExpr` -- Dictionary literals

```cpp
class DictExpr : public Expr {
public:
    std::vector<std::pair<std::unique_ptr<Expr>, std::unique_ptr<Expr>>> entries;   // (key, value) pairs
};
```

Each entry is a key-value pair. An empty `DictExpr` represents `{}` (when no colon is present in the first expression, it is parsed as a set instead).

**Examples:** `{"a": 1, "b": 2}`, `{}`

### `SetExpr` -- Set literals

```cpp
class SetExpr : public Expr {
public:
    std::vector<std::unique_ptr<Expr>> elements;   // The set elements
};
```

Parsed when `{expr, ...}` is seen without colons. Disambiguated from `DictExpr` by the absence of `:` after the first expression.

**Examples:** `{1, 2, 3}`, `{"a", "b"}`

### `ListCompExpr` -- List comprehensions

```cpp
class ListCompExpr : public Expr {
public:
    std::unique_ptr<Expr> element;          // The expression to produce each element
    std::string varName;                     // The loop variable name
    std::unique_ptr<Expr> iterable;         // The iterable being looped over
    std::unique_ptr<Expr> condition;        // Optional filter condition (nullptr if absent)
};
```

**Examples:** `[x*2 for x in items]`, `[x for x in items if x > 0]`

Note that the current implementation supports only a single loop variable (`varName` is a string, not a vector). Tuple unpacking in list comprehensions is not yet supported.

### `DictCompExpr` -- Dictionary comprehensions

```cpp
class DictCompExpr : public Expr {
public:
    std::unique_ptr<Expr> key;                  // The key expression
    std::unique_ptr<Expr> value;                // The value expression
    std::vector<std::string> varNames;          // Loop variable names (supports tuple unpacking)
    std::unique_ptr<Expr> iterable;             // The iterable
    std::unique_ptr<Expr> condition;            // Optional filter (nullptr if absent)
};
```

Unlike `ListCompExpr`, `DictCompExpr` supports multiple variable names for tuple unpacking: `{k: v for k, v in items}`.

**Examples:** `{k: v for k, v in pairs}`, `{x: x**2 for x in range(10) if x > 0}`

### `SetCompExpr` -- Set comprehensions

```cpp
class SetCompExpr : public Expr {
public:
    std::unique_ptr<Expr> element;          // The expression to produce each element
    std::string varName;                     // The loop variable name
    std::unique_ptr<Expr> iterable;         // The iterable being looped over
    std::unique_ptr<Expr> condition;        // Optional filter condition (nullptr if absent)
    std::vector<CompClause> extraClauses;   // Additional `for`/`if` clauses (nested comprehensions)
};
```

Parsed from `{expr for var in iterable if cond}` (braces without a `:`, distinguishing it from `DictCompExpr`).

**Examples:** `{x*2 for x in items}`, `{x for x in items if x > 0}`

### `GeneratorExpr` -- Generator expressions

```cpp
class GeneratorExpr : public Expr {
public:
    std::unique_ptr<Expr> element;          // The expression to produce each element
    std::string varName;                     // The loop variable name
    std::unique_ptr<Expr> iterable;         // The iterable being looped over
    std::unique_ptr<Expr> condition;        // Optional filter condition (nullptr if absent)
    std::vector<CompClause> extraClauses;   // Additional `for`/`if` clauses (nested comprehensions)
};
```

Parsed from `(expr for var in iterable if cond)`. Currently evaluated eagerly as a list.

**Examples:** `(x*2 for x in items)`, `(x for x in items if x > 0)`

### `LambdaExpr` -- Lambda expressions

```cpp
class LambdaExpr : public Expr {
public:
    struct Parameter {
        std::string name;                      // Parameter name
        std::unique_ptr<TypeExpr> type;        // Type annotation (required in Dragon mode)
        std::unique_ptr<Expr> defaultValue;    // Default value (optional)
    };
    std::vector<Parameter> params;              // Lambda parameters
    std::unique_ptr<TypeExpr> returnType;       // Return type (required in Dragon mode)
    std::unique_ptr<Expr> body;                 // Single-expression body (Python-style lambda)
    std::vector<std::unique_ptr<Stmt>> bodyStmts;  // Block body (Dragon-style lambda)
};
```

Supports two forms:
- **Python-style**: `lambda x, y: x + y` -- `body` is set, `bodyStmts` is empty
- **Dragon-style**: `lambda (x: int, y: int) : int { return x + y }` -- `bodyStmts` is populated, `body` is null

The `Parameter` struct here is distinct from the top-level `Parameter` struct (used by `FunctionDecl`). It lacks `isVarArg`/`isKwArg` flags.

### `IfExpr` -- Ternary conditional expression

```cpp
class IfExpr : public Expr {
public:
    std::unique_ptr<Expr> condition;    // The test condition
    std::unique_ptr<Expr> thenExpr;     // Value if condition is true
    std::unique_ptr<Expr> elseExpr;     // Value if condition is false
};
```

Represents Python's ternary syntax: `value_if_true if condition else value_if_false`. Note the field ordering: `thenExpr` is the value (parsed first), `condition` is the test (parsed second), `elseExpr` is the alternative (parsed third).

**Examples:** `x if x > 0 else -x`, `"yes" if flag else "no"`

### `AwaitExpr` -- Await expression

```cpp
class AwaitExpr : public Expr {
public:
    std::unique_ptr<Expr> operand;   // The awaited coroutine/expression
};
```

**Example:** `await fetch_data()`

### `FireExpr` -- Fire (green-thread spawn) expression

```cpp
class FireExpr : public Expr {
public:
    std::unique_ptr<Expr> operand;                      // CallExpr for the `fire fn()` form
    std::vector<std::unique_ptr<Stmt>> bodyStmts;       // for the `fire { block }` form
    std::vector<std::string> capturedVars;              // spec-27: variables captured from enclosing scope
    std::vector<std::string> mutatedCapturedVars;       // spec-27.1: captured vars marked `nonlocal`
};
```

Represents spawning a green-thread vthread (the default concurrency tier). Has two forms: `fire fn(args)` populates `operand` with the call; `fire { block }` populates `bodyStmts`. Either form yields a `Task[T]` handle. The capture vectors mirror the closure machinery used by `LambdaExpr` and `ThreadStmt`.

**Examples:** `t: Task[int] = fire work()`, `fire { do_background() }`

### `YieldExpr` -- Yield expression

```cpp
class YieldExpr : public Expr {
public:
    std::unique_ptr<Expr> value;      // The yielded value (optional, nullptr for bare yield)
    bool isYieldFrom = false;          // true for "yield from", false for "yield"
};
```

**Examples:** `yield 42`, `yield`, `yield from iterable`

### `StarredExpr` -- Starred/unpacking expression

```cpp
class StarredExpr : public Expr {
public:
    std::unique_ptr<Expr> value;       // The expression being starred
    bool isDoubleStar = false;          // true for **kwargs, false for *args
};
```

**Examples:** `*args` (isDoubleStar=false), `**kwargs` (isDoubleStar=true)

## 6. Statement Nodes (22 nodes)

All statement nodes extend `Stmt`.

### `ExprStmt` -- Expression statement

```cpp
class ExprStmt : public Stmt {
public:
    std::unique_ptr<Expr> expr;   // The expression
};
```

A bare expression used as a statement. Common for function calls: `print("hello")`.

### `AssignStmt` -- Simple assignment

```cpp
class AssignStmt : public Stmt {
public:
    std::vector<std::unique_ptr<Expr>> targets;    // Assignment targets (supports chained: a = b = c)
    std::unique_ptr<Expr> value;                    // The value being assigned
    std::unique_ptr<TypeExpr> typeAnnotation;       // Optional type annotation (x: int = 5)
};
```

The `targets` vector supports chained assignment: in `a = b = 5`, both `a` and `b` would be targets. In the current parser implementation, only a single target is produced (the vector always has one element).

**Examples:** `x = 5`, `a = b = 10`

### `AugAssignStmt` -- Augmented assignment

```cpp
class AugAssignStmt : public Stmt {
public:
    std::unique_ptr<Expr> target;    // The target variable
    Token op;                         // The operator token (+=, -=, *=, etc.)
    std::unique_ptr<Expr> value;     // The right-hand value
};
```

The `op` token type is one of: `PLUS_EQUAL`, `MINUS_EQUAL`, `STAR_EQUAL`, `SLASH_EQUAL`, `DOUBLE_SLASH_EQUAL`, `PERCENT_EQUAL`, `POWER_EQUAL`, `AT_EQUAL`, `AMPERSAND_EQUAL`, `PIPE_EQUAL`, `CARET_EQUAL`, `LEFT_SHIFT_EQUAL`, `RIGHT_SHIFT_EQUAL`.

**Examples:** `x += 1`, `total *= 2`, `flags |= mask`

### `AnnAssignStmt` -- Annotated assignment

```cpp
class AnnAssignStmt : public Stmt {
public:
    std::unique_ptr<Expr> target;           // The variable

The `elifClauses` field is a vector of `(condition, body)` pairs. Each elif clause is its own condition and block. The `elseBody` is empty (not null) when there is no else clause.

Represents `thread { block }` - the scoped OS-thread concurrency tier. `thread` is a contextual keyword; the block compiles to a pthread that auto-joins (RAII) at scope exit. The capture vectors mirror the closure machinery used by `FireExpr` and `LambdaExpr`.

```

Represents structural pattern matching (PEP 634): `match subject { case pattern: ... }`. **Example:**

## 7. Declaration Nodes (3 nodes)

Declaration nodes extend `Stmt`, so they can appear in any statement context. They represent function, class, and type-alias definitions.

### `FunctionDecl` -- Function definition

```cpp
class FunctionDecl : public Stmt {
public:
    std::string name;                                   // Function name
    std::vector<TypeParam> typeParams;                  // spec-44: PEP 695 generics; empty = non-generic
    std::vector<Parameter> params;                      // Parameters (see Parameter struct below)
    std::unique_ptr<TypeExpr> returnType;               // Return type annotation (nullptr if omitted)
    std::vector<std::unique_ptr<Stmt>> body;            // Function body statements
    std::vector<std::unique_ptr<Expr>> decorators;      // Decorator expressions
    bool isAsync = false;                                // true for async def
    bool isStatic = false;                               // true for static / @staticmethod
    bool isExtern = false;                               // true for extern "C" def (FFI, no body)
    std::string externSymbol;                            // C linker symbol when name is an alias
    std::string genericHomeModule;                       // spec-44: defining module of a stamped instantiation
    // ... plus method/property flags, overload bookkeeping, captured-var vectors, docstring
};
```

`typeParams` carries the function's PEP 695 generic parameters: the `T` in `def f[T](...)`. A non-empty `typeParams` marks the decl as a *generic template* (the design spec) - it is abstractly type-checked once but never lowered to LLVM directly; only the monomorphizer's stamped instantiations reach CodeGen. User-defined generics are therefore first-class in the node shape, not a desugaring. `genericHomeModule` records the defining module of a stamped cross-module instantiation so its body's bare names still resolve. The full node also tracks `isMethod`, `hasImplicitSelf`, `isClassMethod`, `isConstructor`, `isProperty`, the design spec method-overload indices, positional-only / keyword-only boundaries, captured-variable vectors, and an optional `docstring`.

**Dragon example:**
```
@decorator
def add(x: int, y: int) -> int {
    return x + y
}
```

**Python example:**
```python
@decorator
async def fetch(url: str) -> str:
    return await get(url)
```

### `ClassDecl` -- Class definition

```cpp
class ClassDecl : public Stmt {
public:
    std::string name;                                                          // Class name
    std::vector<TypeParam> typeParams;                                         // spec-44: PEP 695 generics; empty = non-generic
    std::vector<std::unique_ptr<Expr>> bases;                                  // Base class expressions
    std::vector<std::pair<std::string, std::unique_ptr<Expr>>> keywords;       // Keyword args (metaclass, etc.)
    std::vector<std::unique_ptr<Stmt>> body;                                   // Class body
    std::vector<std::unique_ptr<Expr>> decorators;                             // Decorator expressions
    std::optional<std::string> docstring;                                      // first ExprStmt(StringLiteral), powers __doc__
    std::string genericHomeModule;                                             // spec-44: defining module of a stamped instantiation
};
```

The `bases` vector contains expressions for each base class. The `keywords` vector is for keyword arguments like `metaclass=ABCMeta` (declared in the node but not yet extracted by the parser -- they would currently appear in `bases`). As with `FunctionDecl`, `typeParams` carries the class's PEP 695 generic parameters (the `T` in `class Foo[T]`) and `genericHomeModule` records the defining module of a stamped cross-module instantiation - user-defined generics are first-class in the node shape.

**Example:**
```
@dataclass
class Point(Base) {
    x: int = 0
    y: int = 0
}
```

### `TypeAliasStmt` -- Type alias definition

```cpp

```

Represents a PEP 695 type alias: `type Point = tuple[int, int]`. Grouped with the declarations because it introduces a name and, like `FunctionDecl` / `ClassDecl`, is dispatched in the visitor's "Declarations" section.

**Example:** `type Vector = list[float]`

## 8. The Parameter Struct

```cpp
struct Parameter {
    std::string name;                       // Parameter name
    std::unique_ptr<TypeExpr> type;         // Type annotation (required in Dragon, optional in Python)
    std::unique_ptr<Expr> defaultValue;     // Default value expression (optional, nullptr if absent)
    bool isVarArg = false;                  // true for *args parameters
    bool isKwArg = false;                   // true for **kwargs parameters
};
```

This is a top-level struct in `namespace dragon`, used by `FunctionDecl` for function parameters. It is separate from `LambdaExpr::Parameter` (which lacks `isVarArg`/`isKwArg` flags).

**Examples:**
- `x: int` -- `{name="x", type=NamedTypeExpr("int"), defaultValue=null, isVarArg=false, isKwArg=false}`
- `y: str = "default"` -- `{name="y", type=NamedTypeExpr("str"), defaultValue=StringLiteral("default"), ...}`
- `*args: int` -- `{name="args", type=NamedTypeExpr("int"), isVarArg=true}`
- `**kwargs` -- `{name="kwargs", type=null, isKwArg=true}`
- `*` (bare star) -- `{name="", isVarArg=true}` (name is empty, used as separator only)

### The `TypeParam` struct

```cpp
struct TypeParam {
    std::string name;                  // The type variable name (the `T` in `Foo[T]`)
    std::unique_ptr<TypeExpr> bound;   // Reserved for spec-46 (bounded `T: Base`); always nullptr in v1
};
```

A top-level struct in `namespace dragon`, used by both `FunctionDecl` and `ClassDecl` for their PEP 695 generic parameters (the design spec). `bound` is reserved for the future bounded-generics work (spec-46) and is always `nullptr` today; keeping the field present holds the surface stable so spec-46 does not churn it.

## 10. The Visitor Pattern

Dragon's AST uses the classic **double dispatch visitor pattern**. Each AST node implements `accept(ASTVisitor&)`, which calls the appropriate `visit()` overload on the visitor.

### `ASTVisitor` -- Pure virtual visitor base

```cpp

public:
    virtual ~ASTVisitor() = default;

    // 6 type expression methods
    virtual void visit(NamedTypeExpr& node) = 0;
    virtual void visit(GenericTypeExpr& node) = 0;

    virtual void visit(UnionTypeExpr& node) = 0;
    virtual void visit(CallableTypeExpr& node) = 0;
    virtual void visit(TupleTypeExpr& node) = 0;

    // 30 expression methods
    virtual void visit(IntegerLiteral& node) = 0;
    virtual void visit(FloatLiteral& node) = 0;
    virtual void visit(StringLiteral& node) = 0;
    virtual void visit(BooleanLiteral& node) = 0;
    virtual void visit(NoneLiteral& node) = 0;
    virtual void visit(NameExpr& node) = 0;

    virtual void visit(AssignStmt& node) = 0;

    virtual void visit(IfStmt& node) = 0;

    virtual void visit(ForStmt& node) = 0;
    virtual void visit(TryStmt& node) = 0;
    virtual void visit(WithStmt& node) = 0;
    virtual void visit(ThreadStmt& node) = 0;
    virtual void visit(MatchStmt& node) = 0;
    virtual void visit(ReturnStmt& node) = 0;
    virtual void visit(RaiseStmt& node) = 0;
    virtual void visit(BreakStmt& node) = 0;
    virtual void visit(ContinueStmt& node) = 0;

    virtual void visit(DeleteStmt& node) = 0;

```

**Total: 62 pure virtual methods.** Any class that extends `ASTVisitor` must implement all 62. This ensures compile-time checking that every node type is handled.

### How `accept()` works

Each concrete node implements `accept()` in `src/AST.cpp`:

```cpp
void IntegerLiteral::accept(ASTVisitor& visitor) { visitor.visit(*this); }
void BinaryExpr::accept(ASTVisitor& visitor)     { visitor.visit(*this); }
void IfStmt::accept(ASTVisitor& visitor)         { visitor.visit(*this); }
// ... etc for all 61 concrete node types + Module
```

The double dispatch mechanism: calling `node->accept(visitor)` invokes the node's `accept()`, which calls `visitor.visit(*this)`. Because `*this` has a concrete type, the compiler selects the correct `visit()` overload.

### `DefaultASTVisitor` -- Recursive tree walker

```cpp
class DefaultASTVisitor : public ASTVisitor { ... };
```

Implements all 62 visitor methods with a default behavior: **recursively visit all child nodes**. Leaf nodes (literals, `NameExpr`, `BreakStmt`, `ContinueStmt`, `PassStmt`, `GlobalStmt`, `NonlocalStmt`, `ImportStmt`, `FromImportStmt`) have empty implementations (no children to visit).

Compound nodes visit their children in a natural order. For example:

```cpp
void DefaultASTVisitor::visit(BinaryExpr& node) {
    node.left->accept(*this);
    node.right->accept(*this);
}

void DefaultASTVisitor::visit(IfStmt& node) {
    node.condition->accept(*this);
    for (auto& s : node.thenBody) s->accept(*this);
    for (auto& [cond, body] : node.elifClauses) {
        cond->accept(*this);
        for (auto& s : body) s->accept(*this);
    }
    for (auto& s : node.elseBody) s->accept(*this);
}

void DefaultASTVisitor::visit(FunctionDecl& node) {
    for (auto& dec : node.decorators) dec->accept(*this);
    for (auto& p : node.params) {
        if (p.type) p.type->accept(*this);
        if (p.defaultValue) p.defaultValue->accept(*this);
    }
    if (node.returnType) node.returnType->accept(*this);
    for (auto& s : node.body) s->accept(*this);
}
```

Subclasses of `DefaultASTVisitor` can override only the methods they care about and rely on the default implementation for tree traversal.

### `ASTPrinter` -- S-expression output for debugging

```cpp
class ASTPrinter : public ASTVisitor {
public:
    std::string print(ASTNode& node);  // Entry point: returns the formatted string
private:
    std::string output_;    // Accumulated output text
    int indent_ = 0;        // Current indentation level

    void write(const std::string& text);           // Append without indentation/newline

Produces a Lisp-like S-expression representation of the AST. Used by `dr check --dump-ast`. The output format uses 2-space indentation.

**Example output for `x = 1 + 2`:**

```

Each visit method is responsible for printing its own node and recursively visiting children. Indentation is managed with `increaseIndent()`/`decreaseIndent()` calls that bracket child traversal.

## 11. Memory Model

### Ownership with `unique_ptr`

The AST uses `std::unique_ptr` throughout for ownership. Each parent node owns its children:

- `Module` owns its `vector<unique_ptr<Stmt>> body`
- `BinaryExpr` owns its `unique_ptr<Expr> left` and `unique_ptr<Expr> right`
- `FunctionDecl` owns its `vector<unique_ptr<Stmt>> body`, `vector<unique_ptr<Expr>> decorators`, etc.

This means the AST is a strict tree (no shared ownership, no cycles). Destroying the root `Module` node recursively destroys the entire tree.

### Node creation

Nodes are created using `std::make_unique<NodeType>()` in the parser and then populated field by field:

```cpp
auto lit = std::make_unique<IntegerLiteral>();
lit->setLocation(previous().location());
lit->value = 42;
return lit;  // Ownership transferred to caller via unique_ptr
```

Fields are set directly (public members), not through constructors. Most AST nodes have no user-defined constructors -- they rely on default construction followed by field assignment.

### The `shared_ptr<Type>` exception

The `Expr::type` field uses `shared_ptr<Type>`, not `unique_ptr`. This is because the type system has shared types: many expressions can have the same `int` type, and type objects are shared across the entire AST rather than being owned by individual nodes. The type checker creates type objects and attaches them via `shared_ptr`.

### Strings are owned copies

Both `NameExpr::name` and `StringLiteral::value` (and all other string fields) use `std::string`, not `std::string_view`. This avoids lifetime issues -- the token stream that provided the original lexemes may be destroyed before the AST is done being used.

### Token objects in AST nodes

`BinaryExpr::op`, `UnaryExpr::op`, and `AugAssignStmt::op` store full `Token` objects (not pointers or references). The `Token` class stores its lexeme as `std::string`, so these are value copies that are self-contained.

## 13. Nested Helper Structs

Several AST nodes contain nested struct definitions:

### `TryStmt::ExceptHandler`

```cpp
struct ExceptHandler {
    std::unique_ptr<TypeExpr> type;     // Exception type (optional)
    std::string name;                    // Binding name (optional, empty = unbound)
    std::vector<std::unique_ptr<Stmt>> body;
};
```

### `WithStmt::WithItem`

```cpp
struct WithItem {
    std::unique_ptr<Expr> contextExpr;     // The context manager expression
    std::unique_ptr<Expr> optionalVars;    // The 'as' target (optional)
};
```

### `ImportStmt::Alias`

```cpp
struct Alias {
    std::string name;       // Module or symbol name
    std::string asName;     // Alias (empty if no 'as' clause)
};
```

Reused by `FromImportStmt` for its imported names.

### `LambdaExpr::Parameter`

```cpp
struct Parameter {
    std::string name;
    std::unique_ptr<TypeExpr> type;        // Optional type annotation
    std::unique_ptr<Expr> defaultValue;    // Optional default value
};
```

Distinct from the top-level `dragon::Parameter` struct (which adds `isVarArg`/`isKwArg`).

## 14. How the Type Checker Annotates the AST

The `Expr::type` field is a `std::shared_ptr<Type>` that starts as `nullptr`. During type checking, the TypeChecker visitor traverses the AST and sets this field on every expression node.

For example, after type checking:
- `IntegerLiteral` nodes get `type = PrimitiveType("int")`
- `BooleanLiteral` nodes get `type = PrimitiveType("bool")`
- `BinaryExpr` with `+` between two `int` operands gets `type = PrimitiveType("int")`
- `ListExpr` with all `int` elements gets `type = ListType(PrimitiveType("int"))`
- `CallExpr` calling a function returning `str` gets `type = PrimitiveType("str")`

The type annotation on `Expr` is the bridge between the parser and the code generation backend. The CodeGen reads `expr->type` to determine what LLVM IR to generate.

Statement nodes (`Stmt`) and type expression nodes (`TypeExpr`) do not have a `type` field -- only `Expr` nodes carry inferred types.

## 15. Visitors in the Codebase

The following classes implement `ASTVisitor`:

| Visitor | File | Purpose |
|---------|------|---------|
| `DefaultASTVisitor` | `src/AST.cpp` | Recursive walk with no-op defaults |
| `ASTPrinter` | `src/AST.cpp` | S-expression dump for `--dump-ast` |
| `Sema` (internal) | `src/Sema.cpp` | Semantic analysis (name resolution, scope) |
| `TypeChecker` (internal) | `src/TypeChecker.cpp` | Type inference and validation |
| `CodeGen` (internal) | `src/CodeGen.cpp` | LLVM IR code generator |

All visitor classes implement all 62 pure virtual methods. The pimpl pattern means the actual visitor implementation is often inside a nested `Impl` struct that extends `ASTVisitor`.
